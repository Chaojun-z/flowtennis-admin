# FlowTennis 订场会员模块重构规格

> 本文档用于修正当前会员模块“能跑但不好用、对象边界不清、规则难理解”的问题。  
> 后续开发会员模块时，必须先读本文档，再读：
> - `_local/requirements/2026-04-12-membership-management-implementation-plan.md`
> - `_local/requirements/2026-04-12-teaching-entitlement-unified-development-plan.md`

---

## 1. 这份文档解决什么问题

当前会员模块的主要问题不是“少几个字段”，而是：

1. 会员方案、会员购买、会员账户、赠送权益、权益流水这 5 个对象没有被清楚地区分。
2. 用户在“编辑订场用户”时，容易误触发会员购买，导致重复生成购买记录。
3. 会员方案没有把赠送权益模板结构化配置完整。
4. 会员时间规则、折扣规则、续充规则没有在界面上明确展示。
5. 权益消耗没有明确入口，运营不知道该在哪里扣减。
6. 当前页面按“表”组织，而不是按“业务动作”组织，理解成本高。

本次重构目标：

- 把会员模块整理成一套运营看得懂、解释成本低、开发边界清晰的系统。
- 保证一个业务动作只生成一类数据，不允许“改资料顺带买会员”。
- 保证余额、折扣、赠送权益、权益消耗各走各的账，不混账。

## 2. 最终业务定义

### 2.1 会员方案是什么

会员方案 = 可售卖的标准模板。

它定义：

- 充值档位
- 当前档位折扣
- 赠送金额
- 赠送权益模板
- 默认规则说明

它不代表任何真实成交。

### 2.2 会员购买记录是什么

会员购买记录 = 某个订场用户在某一天做的一次开卡/续充成交记录。

它回答的是：

- 谁买了
- 买的是哪个方案
- 实际充值多少
- 当次赠送了什么
- 当次权益有效到什么时候

一个用户可以有多条购买记录。

### 2.3 会员账户是什么

会员账户 = 某个订场用户当前正在生效的会员状态总览。

它回答的是：

- 当前是不是会员
- 当前会员标签是什么
- 当前订场折扣是多少
- 当前余额有效到什么时候
- 当前余额最晚什么时候必须清零
- 最近一次续充是哪次

一个订场用户只有一个会员账户主视图。

### 2.4 赠送权益批次是什么

赠送权益批次 = 每次会员购买记录生成的一包独立赠送权益。

比如：

- 2026-04-12 充值 5000，送 6 次发球机、6 次大师公开课
- 2026-08-01 再充 2000，再送 2 次发球机、2 次大师公开课

这两批权益必须分别存在，分别到期。

### 2.5 权益流水是什么

权益流水 = 赠送权益的使用、补发、作废、过期失效记录。

它回答的是：

- 哪一批权益被用了
- 用了几次
- 谁操作的
- 为什么操作
- 什么时候操作的

结论：

```text
会员方案 = 商品模板
会员购买记录 = 成交记录
会员账户 = 当前状态
赠送权益批次 = 每次购买送了什么
权益流水 = 这些赠送权益后来怎么被消耗/补发/失效
```

## 3. 最终业务规则

## 3.1 余额规则

会员余额仍然以 `courts.history` 为唯一资金主账。

规则如下：

1. 每次会员充值都会向订场用户余额充值。
2. 余额有效期按最近一次合规续充日期重新计算。
3. 余额最长有效期 2 年。
4. 到 2 年后剩余余额自动清零。
5. 清零只影响余额，不补生成消费，不改历史购买记录。

## 3.2 折扣规则

折扣和赠送权益分开。

最终规则：

1. 用户当前订场折扣 = 最近一次充值档位对应折扣。
2. 不保留历史折扣叠加。
3. 续充哪个档位，就按哪个档位折扣执行。
4. 折扣只影响后续订场消费，不回改历史消费。

## 3.3 赠送权益规则

赠送权益和余额分开。

最终规则：

1. 每次充值都会生成一批独立赠送权益。
2. 每批赠送权益有效期 12 个月。
3. 不同批次分别到期，不合并有效期。
4. 消耗时优先扣最早到期的批次。
5. 过期后自动失效，但保留记录。
6. 赠送权益不能折现，不能并入余额。

## 3.4 降档续充规则

最终规则：

1. 续充 5000，折扣按 5000 档位。
2. 后续改充 2000，折扣改按 2000 档位。
3. 原来 5000 档位送的其他权益继续保留，直到各自到期。
4. 折扣权益和其他赠送权益绝对分开处理。

## 4. 推荐会员权益类型

本期不要以“自由文本权益”为主，应该以结构化字段为主。

会员方案固定权益字段定义如下：

1. `bonusAmount`
   - 赠送金额
   - 单位：元

2. `publicLessonCount`
   - 大师公开课
   - 单位：次

3. `stringingLaborCount`
   - 穿线免手工费
   - 单位：次

4. `ballMachineCount`
   - 发球机免费
   - 单位：次

5. `level2PartnerCount`
   - 国家二级运动员陪打
   - 单位：次

6. `designatedCoachPartnerCount`
   - 指定教练陪打
   - 单位：次

7. `designatedCoachIds`
   - 指定教练范围
   - 单位：教练 ID 数组

8. `notes`
   - 备注
   - 用于记录补充说明

如确实需要扩展，再保留：

- `customBenefits[]`

但只能作为补充字段，不能替代固定字段。

## 5. 页面结构重构建议

## 5.1 一级入口保留“会员管理”

保留一个统一模块没问题，但不能把所有表平铺成同级 tab 并让运营自己理解。

### 会员管理下拆成 4 个视角

1. **会员方案**
   - 看的是卖什么
   - 负责创建、停用、查看方案规则

2. **会员购买**
   - 看的是谁买了什么
   - 负责开卡、续充、查看成交记录

3. **会员账户**
   - 看的是当前谁是会员、当前折扣、余额有效期
   - 负责看当前状态，不负责新增交易

4. **赠送权益**
   - 看的是送了什么、还剩什么、用了什么
   - 负责消耗、补发、查看流水

结论：

```text
一个模块可以
一个页面也可以
但页面内部必须按业务视角分层，而不是只按数据库表分 tab
```

## 5.2 高频操作入口不在会员管理，而在订场用户详情

日常最常用的动作应该都放到订场用户详情里：

1. 开卡/续充会员
2. 查看当前会员状态
3. 查看余额有效期 / 最晚清零时间
4. 查看赠送权益剩余
5. 查看最近购买记录
6. 执行权益消耗

也就是说：

- `会员管理` 是后台运营总览页
- `订场用户详情` 才是高频操作页

## 6. 关键页面设计

## 6.1 会员方案页

目标：管理标准档位。

列表字段建议：

- 方案名称
- 充值金额
- 折扣
- 赠送金额
- 大师公开课
- 穿线免手工费
- 发球机免费
- 国家二级运动员陪打
- 指定教练陪打
- 余额规则摘要
- 权益规则摘要
- 状态

新建/编辑表单字段建议：

- 方案名称
- 充值金额
- 折扣
- 赠送金额
- 大师公开课次数
- 穿线免手工费次数
- 发球机免费次数
- 国家二级运动员陪打次数
- 指定教练陪打次数
- 指定教练范围
- 余额规则展示
- 赠送权益规则展示
- 备注

页面必须直接展示以下规则文案：

- 余额续充后统一按最新充值日期重新计算最长 2 年
- 当次赠送权益独立生效 12 个月
- 当前折扣按最近一次充值档位执行

## 6.2 会员购买页

目标：管理真实成交记录。

列表字段建议：

- 购买日期
- 订场用户
- 会员方案
- 实充金额
- 赠送金额
- 当前折扣
- 当次权益有效期
- 余额重置到期日
- 操作人
- 状态

新建动作必须改名为：

- `开卡/续充会员`

不能叫“新增会员购买”，也不能在普通资料编辑里顺带生成。

弹窗字段建议：

- 订场用户
- 会员方案
- 成交日期
- 实充金额
- 折扣快照
- 赠送金额快照
- 当次赠送权益快照
- 本次余额有效期规则展示
- 本次赠送权益有效期规则展示
- 操作备注

保存前弹窗必须二次确认：

- 本次将生成 1 条会员购买记录
- 本次将写入 1 条订场充值流水
- 本次将更新会员账户当前折扣和有效期
- 本次将生成 1 批赠送权益

## 6.3 会员账户页

目标：看当前状态，不做交易。

列表字段建议：

- 订场用户
- 当前会员标签
- 当前折扣
- 当前余额
- 余额有效期
- 最晚清零日
- 当前可用赠送权益摘要
- 最近一次续充
- 状态

这里不提供“创建购买记录”的按钮，只提供：

- 查看详情
- 去续充

## 6.4 赠送权益页

目标：管理权益批次和权益流水。

建议分成上下两块：

### 上半区：权益批次

字段：

- 订场用户
- 来源购买记录
- 权益类型
- 总数
- 已用
- 剩余
- 到期日
- 状态

### 下半区：权益流水

字段：

- 日期
- 订场用户
- 权益类型
- 动作（消耗 / 补发 / 失效 / 作废）
- 数量
- 来源批次
- 操作人
- 备注

## 7. 订场用户详情必须新增的会员区块

订场用户详情页必须成为会员高频操作主入口。

### 7.1 顶部会员摘要卡

显示：

- 当前是否会员
- 当前会员标签
- 当前折扣
- 当前余额
- 余额有效期
- 最晚清零时间

按钮：

- `开卡/续充会员`
- `查看购买记录`

### 7.2 赠送权益卡

每种权益单独一行：

- 大师公开课：剩余 X 次
- 穿线免手工费：剩余 X 次
- 发球机免费：剩余 X 次
- 国家二级运动员陪打：剩余 X 次
- 指定教练陪打：剩余 X 次

每行按钮：

- `消耗 1 次`
- `补发`
- `查看流水`

### 7.3 最近购买记录卡

显示最近 3-5 条：

- 日期
- 方案
- 充值金额
- 折扣
- 当次权益摘要

## 8. 权益消耗操作规范

权益消耗必须是一个独立动作，不允许直接改剩余次数。

### 8.1 消耗弹窗字段

- 订场用户
- 权益类型
- 消耗数量
- 关联日期
- 扣减批次（默认自动选择最早到期批次）
- 操作人
- 备注

如果是“指定教练陪打”，额外增加：

- 实际教练

### 8.2 消耗逻辑

1. 系统按权益类型筛出所有未过期且有剩余的批次。
2. 默认先扣最早到期批次。
3. 写入权益流水。
4. 刷新批次剩余数量。

### 8.3 补发逻辑

补发不是改原购买记录，而是新增一条权益流水：

- `action = supplement`

必要时可要求填写原因：

- 运营补发
- 客诉补偿
- 手工纠错

## 9. 防重复和防误操作规则

这是必须实现的底层保护。

### 9.1 编辑订场用户资料不得创建会员购买

订场用户普通资料编辑只允许改：

- 姓名
- 电话
- 备注
- 学员关联

不得顺带触发：

- 新建会员购买记录
- 生成权益批次
- 写会员充值流水

### 9.2 会员购买只能从专门入口创建

只能从以下入口创建：

- 会员购买页的 `开卡/续充会员`
- 订场用户详情里的 `开卡/续充会员`

### 9.3 提交防重

会员购买提交时必须防止重复生成：

- 前端按钮提交后立即禁用
- 后端要求幂等键或至少校验短时间重复提交

### 9.4 历史记录不可回写污染

会员方案后续即使改了字段：

- 老的会员购买记录快照不回写
- 老的权益批次不回写
- 老的流水不回写

## 10. 推荐数据关系

```text
membership_plans
  ↓ 生成快照
membership_orders
  ↓ 更新当前状态
membership_accounts
  ↓ 生成当次赠送权益
membership_benefit_batches
  ↓ 被操作
membership_benefit_ledger
```

说明：

- 目前代码里如果没有单独的 `benefit_batches` 表，建议下一步补出来。
- 如果短期不建新表，也必须在购买记录中有“当次权益批次快照”，并能稳定定位一批权益。
- 从长期可维护性看，`benefit_batches` 独立成表更清晰。

## 11. 开发优先级

## P0 先做

1. 禁止“编辑订场用户资料”创建会员购买
2. 会员购买只从专用入口创建
3. 会员方案补齐结构化权益字段
4. 会员方案页展示余额/权益/折扣规则
5. 订场用户详情增加会员摘要和权益区块
6. 权益消耗入口落地
7. 重复提交防重

## P1 再做

1. 会员管理页内部结构重排
2. 单独的权益批次展示
3. 学员详情增加更完整的会员摘要
4. 更清晰的到期提醒和状态标签

## P2 后做

1. 自动提醒
2. 批量操作
3. 更复杂的运营报表

## 12. 验收标准

以下场景全部通过，才算会员模块可投入使用：

1. 编辑订场用户资料，不会新增会员购买记录。
2. 开卡一次，只生成 1 条购买记录、1 条充值流水、1 批赠送权益。
3. 同一用户续充两次，购买记录是两条，但会员账户仍是一个当前账户。
4. 当前折扣等于最近一次充值档位。
5. 之前赠送权益继续保留并分别到期。
6. 权益消耗时能明确扣减哪一批。
7. 会员方案能直接配置结构化赠送权益。
8. 运营在订场用户详情页能完成高频会员操作。
9. 教练端仍然看不到充值金额、支付方式、会员财务信息。

## 13. 本文档后的执行建议

后续开发时，先按本文档做一次会员模块产品重构，不要继续在现有页面上零散补字段。

推荐顺序：

1. 先改数据对象和动作边界
2. 再改会员方案表单
3. 再改订场用户详情高频入口
4. 最后整理会员管理总览页

如果继续沿当前页面小修小补，后面很容易再次出现：

- 重复购买
- 用户看不懂
- 运营解释成本高
- 规则没法统一
