# 课程产品升级为课包 / 购买 / 权益账户实施计划

> **给后续执行线程：** 本文档是本地交接方案，路径在 `_local/requirements/`，不要提交到 GitHub。执行代码前必须先读本文档和 `2026-04-11-coach-feedback-course-package-requirements.md`。实现时控制范围，不做大重构。

**目标：** 把现有“课程产品 -> 班次 -> 学习计划 -> 排课消课”升级为“售卖课包 -> 学员购买 -> 权益账户 -> 排课选择权益账户 -> 消耗记录”的可追踪模型。

**核心结论：** 排课时采用“系统推荐 + 人工确认 + 可手动更改”。系统推荐最匹配且最快过期的课包，但保存前必须明确绑定权益账户，不能静默自动扣课。

**当前状态判断：**
- 现有课程产品更像“课程模板”，字段主要是名称、类型、价格、课时、人数、备注。
- 现有班次和学习计划已经能同步课时，但不能区分“同一学员买了多个不同课包”。
- 现有排课只通过 `classId + lessonCount` 消耗班次/学习计划课时，没有独立购买记录、权益账户和消耗流水。
- 本次不能继续把价格、有效期、活动期、权益规则塞进备注。

---

## 1. 需求边界

### 1.1 本期必须做

1. 新增“售卖课包”模型。
   - 字段：名称、课程类型、价格、总课时、有效天数、购买开始日期、购买结束日期、可用教练、可用校区、可使用日期范围、每日可使用时间段、时段类型、适用人数、状态、备注。
   - 例子：五一私教课包，1000 元 5 节，有效 60 天，活动期 2026-05-01 至 2026-05-05，可用教练朝珺，可用校区马坡，只能在非黄金时段使用。

2. 新增“购买记录”模型。
   - 字段：学员、售卖课包、购买日期、实收金额、支付方式、销售/经办人、备注、状态。
   - 一条购买记录必须生成一个权益账户。

3. 新增“权益账户”模型。
   - 字段：学员、购买记录、售卖课包、课程类型、绑定教练限制、校区限制、可使用日期范围、每日可使用时间段、时段类型、总课时、已用课时、剩余课时、有效期开始、有效期结束、状态。
   - 同一个学员可以同时有多个权益账户。

4. 新增“消耗记录”模型。
   - 字段：权益账户、排课、学员、消耗课时、消耗原因、操作类型、操作人、创建时间。
   - 每次排课扣课必须有消耗记录。
   - 取消排课必须产生反向退回记录，不能只改余额数字。

5. 排课时选择权益账户。
   - 选择学员后展示该学员可用权益账户。
   - 系统默认推荐一个最合适课包。
   - 管理员可以手动改选。
   - 保存排课前必须确认消耗哪个权益账户。

6. 学员详情展示课包。
   - 展示该学员所有权益账户：课包名、课程类型、剩余/总课时、有效期、状态。
   - 能查看消耗明细：哪天哪节课扣了哪个课包。

7. 管理端展示和维护。
   - 课程产品页保留基础模板职责。
   - 新增或扩展“课包/购买/权益账户”管理入口。
   - 管理员可查看购买记录、权益余额和消耗记录。

8. 教练端同步。
   - 教练端看自己的课表、学员、反馈。
   - 教练端不显示课包价格、实收金额、支付方式。
   - 教练端可以看到课程类型、课包名或“剩余课时”这类非财务信息，是否展示以页面简洁为准。

### 1.2 本期不做

1. 不做复杂用户端。
2. 不做退款完整财务闭环。
3. 不做多销售分佣。
4. 不做课包转让给其他学员。
5. 不做组合课包，例如一个课包同时含私教 5 节和团课 10 节。
6. 不做复杂审批流。
7. 不做海报、成长体系、约球功能。
8. 不把历史班次课时强制迁移成权益账户，除非单独做迁移方案。

## 2. 推荐数据模型

### 2.1 课程产品 `products`

继续保留，作为基础课程模板。

建议含义：
- `name`：成人私教、青少年团课、训练营。
- `type`：私教课、团课、训练营等。
- `maxStudents`：适用人数。
- `notes`：基础说明。

不建议继续把活动价、有效期、活动期放在这里。

### 2.2 售卖课包 `packages`

新增表，建议后端表名：

`ft_packages`

建议字段：

```json
{
  "id": "pkg-uuid",
  "name": "五一私教课包",
  "productId": "prod-private-lesson",
  "productName": "成人私教",
  "courseType": "私教课",
  "price": 1000,
  "lessons": 5,
  "validDays": 60,
  "saleStartDate": "2026-05-01",
  "saleEndDate": "2026-05-05",
  "usageStartDate": "2026-05-02",
  "usageEndDate": "2026-07-01",
  "dailyTimeWindows": [
    {
      "label": "非黄金时段",
      "startTime": "07:00",
      "endTime": "17:00",
      "daysOfWeek": [1, 2, 3, 4, 5]
    }
  ],
  "timeBand": "非黄金时段",
  "coachIds": ["coach-a"],
  "coachNames": ["朝珺"],
  "campusIds": ["mabao"],
  "maxStudents": 1,
  "status": "active",
  "notes": "五一活动课包",
  "createdAt": "2026-04-12T00:00:00.000Z",
  "updatedAt": "2026-04-12T00:00:00.000Z"
}
```

规则：
- `coachIds` 为空代表不限教练。
- `campusIds` 为空代表不限校区。
- `validDays` 从购买日期计算有效期。
- `saleStartDate` / `saleEndDate` 是“可购买时间”，只控制什么时候能买。
- `usageStartDate` / `usageEndDate` 是“可上课消耗日期”，控制这个课包能在哪些日期上课。
- `dailyTimeWindows` 是“每天可上课消耗时间段”，用于区分黄金时段、非黄金时段、周末时段等。
- `daysOfWeek` 使用 1-7 表示周一到周日；为空代表每天都可用。
- `timeBand` 是展示标签，例如 `黄金时段`、`非黄金时段`、`全天`；真正校验以后端 `dailyTimeWindows` 为准。
- `status=inactive` 的课包不能继续新购买，但历史权益账户仍保留。

### 2.3 购买记录 `purchases`

新增表：

`ft_purchases`

建议字段：

```json
{
  "id": "pur-uuid",
  "studentId": "stu-a",
  "studentName": "张三",
  "packageId": "pkg-uuid",
  "packageName": "五一私教课包",
  "purchaseDate": "2026-05-02",
  "amountPaid": 1000,
  "payMethod": "微信",
  "operator": "Admin John",
  "status": "active",
  "notes": "",
  "createdAt": "2026-04-12T00:00:00.000Z",
  "updatedAt": "2026-04-12T00:00:00.000Z"
}
```

规则：
- 新增购买记录时，同步创建一个权益账户。
- 购买日期必须在课包活动购买时间内；如果不在范围内，第一版直接禁止保存。
- `amountPaid` 默认取售卖课包价格，但允许管理员手动改为实际实收。

### 2.4 权益账户 `entitlements`

新增表：

`ft_entitlements`

建议字段：

```json
{
  "id": "ent-uuid",
  "studentId": "stu-a",
  "studentName": "张三",
  "purchaseId": "pur-uuid",
  "packageId": "pkg-uuid",
  "packageName": "五一私教课包",
  "productId": "prod-private-lesson",
  "courseType": "私教课",
  "totalLessons": 5,
  "usedLessons": 2,
  "remainingLessons": 3,
  "validFrom": "2026-05-02",
  "validUntil": "2026-07-01",
  "usageStartDate": "2026-05-02",
  "usageEndDate": "2026-07-01",
  "dailyTimeWindows": [
    {
      "label": "非黄金时段",
      "startTime": "07:00",
      "endTime": "17:00",
      "daysOfWeek": [1, 2, 3, 4, 5]
    }
  ],
  "timeBand": "非黄金时段",
  "coachIds": ["coach-a"],
  "coachNames": ["朝珺"],
  "campusIds": ["mabao"],
  "maxStudents": 1,
  "status": "active",
  "createdAt": "2026-04-12T00:00:00.000Z",
  "updatedAt": "2026-04-12T00:00:00.000Z"
}
```

规则：
- `remainingLessons = totalLessons - usedLessons`，后端计算为准。
- 过期判断使用 `validUntil`。
- 使用日期判断优先看 `usageStartDate` / `usageEndDate`，如果为空，则退回使用 `validFrom` / `validUntil`。
- 每日时间段判断使用 `dailyTimeWindows`，为空代表全天可用。
- `status` 可用值建议：`active`、`expired`、`depleted`、`voided`。
- 第一版不允许直接手工改 `usedLessons`，只能通过消耗记录变动。

### 2.5 消耗记录 `entitlement_ledger`

新增表：

`ft_entitlement_ledger`

建议字段：

```json
{
  "id": "led-uuid",
  "entitlementId": "ent-uuid",
  "studentId": "stu-a",
  "scheduleId": "sch-uuid",
  "lessonDelta": -1,
  "action": "consume",
  "reason": "排课消课",
  "operator": "Admin John",
  "createdAt": "2026-04-12T00:00:00.000Z"
}
```

规则：
- 排课保存并消课：写 `lessonDelta=-1`。
- 取消排课退课：写 `lessonDelta=+1`。
- 不物理删除消耗记录。
- 权益账户的 `usedLessons` 由 ledger 结果同步更新，不能只依赖前端。

## 3. 推荐课包选择规则

### 3.1 可用课包筛选

排课时，系统先从该学员权益账户中筛选可用课包：

1. `studentId` 匹配。
2. `status=active`。
3. `remainingLessons >= lessonCount`。
4. 排课日期在 `validFrom` 和 `validUntil` 范围内。
5. 排课日期在 `usageStartDate` 和 `usageEndDate` 范围内；如果这两个字段为空，则使用 `validFrom` 和 `validUntil`。
6. 排课开始/结束时间必须落在 `dailyTimeWindows` 允许范围内；如果为空，则全天可用。
7. `courseType` 与排课课程类型匹配。
8. 如果权益账户限制教练，则排课教练必须在 `coachIds` 或 `coachNames` 内。
9. 如果权益账户限制校区，则排课校区必须在 `campusIds` 内。
10. 如果权益账户限制人数，则排课关联学员人数不能超过 `maxStudents`。

时间限制示例：

```json
{
  "dailyTimeWindows": [
    {
      "label": "非黄金时段",
      "startTime": "07:00",
      "endTime": "17:00",
      "daysOfWeek": [1, 2, 3, 4, 5]
    },
    {
      "label": "周末上午",
      "startTime": "08:00",
      "endTime": "12:00",
      "daysOfWeek": [6, 7]
    }
  ]
}
```

校验口径：
- 排课 `startTime` 和 `endTime` 必须完整落在某一个可用时间段里。
- 不允许一节课跨两个不连续时段，例如 16:30-17:30 不能使用 07:00-17:00 的非黄金时段课包。
- 如果网球场地黄金时段价格不同，第一版建议做成不同售卖课包，例如“私教 5 节非黄金时段”和“私教 5 节黄金时段”，不要在同一个课包里做复杂动态价格。

### 3.2 推荐排序

可用课包超过一个时，系统推荐：

1. 最快过期优先。
2. 使用时间段完全匹配优先，例如排课在非黄金时段，则优先推荐非黄金时段课包。
3. 过期日期相同，剩余课时更少优先。
4. 剩余课时相同，购买时间更早优先。

页面文案示例：

```text
推荐使用：五一私教课包
剩余 3/5 节，有效期至 2026-07-01，非黄金时段可用，匹配教练朝珺、校区马坡。
```

### 3.3 手动更改

管理员可以展开选择其他权益账户。

第一版规则：
- 完全匹配的权益账户可以选择并保存。
- 不匹配或过期的权益账户不允许保存。
- 暂不做“管理员强制使用”，避免财务风险。

后续如果业务确实需要“特殊放行”，再单独加：
- 放行原因必填。
- 只允许 admin。
- 记录审计日志。

## 4. 逻辑串联

### 4.1 新建售卖课包

流程：

```text
课程产品 -> 新建售卖课包 -> 设置价格/课时/有效期/活动期/可使用日期/每日可使用时间段/可用教练/校区 -> 保存
```

影响：
- 不影响历史班次。
- 不影响历史排课。
- 只影响后续购买。

### 4.2 学员购买课包

流程：

```text
学员详情或购买页 -> 选择学员 -> 选择售卖课包 -> 输入实收和支付方式 -> 保存
```

后端动作：
1. 校验课包存在且可购买。
2. 校验购买日期在活动购买范围内。
3. 创建购买记录。
4. 创建权益账户。
5. 返回购买记录和权益账户，前端刷新学员详情。

### 4.3 排课消课

流程：

```text
新增排课 -> 选择学员/班次 -> 选择课程类型/教练/校区/时间 -> 系统推荐权益账户 -> 管理员确认或改选 -> 保存排课
```

后端动作：
1. 校验教练、学员、场地、订场冲突。
2. 校验权益账户可用，包括课程类型、教练、校区、日期、每日时间段、剩余课时。
3. 保存排课记录，排课记录写入 `entitlementId`、`packageName`、`courseType`。
4. 写消耗记录。
5. 更新权益账户已用/剩余课时。
6. 返回排课、权益账户、消耗记录。

### 4.4 编辑排课

触发重新校验的字段：
- 学员。
- 课程类型。
- 教练。
- 校区。
- 上课时间。
- 消课节数。
- 权益账户。
- 状态。

规则：
- 如果只是改备注，不重新扣课。
- 如果从已排课改为已取消，退回原权益账户。
- 如果从已取消改回已排课，重新选择并扣课。
- 如果改了权益账户，先退回旧权益账户，再扣新权益账户。
- 如果已存在课后反馈，不允许随意换学员和权益账户；第一版建议直接禁止，避免历史记录错位。

### 4.5 取消排课

流程：

```text
排课详情 -> 取消排课 -> 后端写退回流水 -> 权益账户课时恢复
```

规则：
- 不建议直接删除已消耗权益的排课。
- 如果已有反馈，禁止删除，只允许取消并保留记录。
- 取消后该课不计入教练工作量和有效消耗。

## 5. 多模块联动

### 5.1 课程产品

变化：
- 继续作为基础模板。
- 不承担活动价和有效期。
- 已被售卖课包引用的课程产品不允许删除。

### 5.2 售卖课包

新增入口：
- 建议在“课程产品”下增加“售卖课包”子区域，或新增单独页“课包管理”。

删除规则：
- 已有购买记录的课包不能删除，只能停用。

### 5.3 学员管理

新增展示：
- 学员详情增加“课包权益”区块。
- 显示：课包名、课程类型、剩余/总课时、有效期、状态。
- 显示消耗明细。

删除规则：
- 学员已有购买记录、权益账户、消耗记录时，不能删除。

### 5.4 班次管理

建议第一版兼容现状：
- 班次仍然可以存在，用来组织团课/长期班。
- 班次可以关联课程产品和教练。
- 排课时如果选择班次，系统根据班次学员、课程类型、教练、校区筛选每个学员的权益账户。

风险点：
- 多学员团课时，每个学员可能消耗不同权益账户。
- 第一版若要控制复杂度，建议团课排课也要求每个学员都有可用权益账户。

### 5.5 学习计划

建议：
- 学习计划继续展示学习进度和班次维度。
- 课时余额逐步从权益账户读取。
- 不再把学习计划作为最终课时余额来源。

兼容策略：
- 第一版先不删除学习计划课时字段。
- 排课保存时，同时维护旧的班次/学习计划课时和新的权益账户课时，保证现有页面不崩。
- 后续再做“学习计划余额显示切换为权益账户汇总”。

### 5.6 排课表

新增字段：

```json
{
  "courseType": "私教课",
  "entitlementId": "ent-uuid",
  "packageName": "五一私教课包",
  "purchaseId": "pur-uuid"
}
```

多学员团课预留字段：

```json
{
  "entitlementUses": [
    {
      "studentId": "stu-a",
      "entitlementId": "ent-a",
      "lessonCount": 1
    },
    {
      "studentId": "stu-b",
      "entitlementId": "ent-b",
      "lessonCount": 1
    }
  ]
}
```

建议：
- 私教一对一可以先用 `entitlementId`。
- 团课多人必须使用 `entitlementUses`，不要把多个学员扣到同一个账户上。

### 5.7 教练运营

展示变化：
- 日/周/月视图课程块可显示课程类型。
- 详情里可显示课包名和剩余课时。
- 不显示价格和实收金额。

统计口径：
- 教练工作量仍基于排课状态。
- 课包消耗统计基于权益账户 ledger。

### 5.8 订场

联动：
- 排课已经要校验订场占用。
- 权益账户不直接影响订场。
- 如果未来售卖“订场课包”，应单独建订场权益，不和上课课包混在第一版。

### 5.9 反馈

## 10. 验收标准

### 10.1 购买和权益

- 同一个学员可以购买多个课包。
- 五一课包和六一课包能同时存在，不互相覆盖。
- 购买五一课包后，学员详情出现一个权益账户。
- 权益账户显示总课时、已用课时、剩余课时、有效期。

### 10.2 排课和消耗

- 排课选择学员后，系统推荐最匹配且最快过期的课包。
- 管理员可以手动改选其他匹配课包。
- 保存排课后，对应权益账户剩余课时减少。
- 取消排课后，对应权益账户剩余课时恢复。
- 课包过期、剩余不足、教练不匹配、校区不匹配、课程类型不匹配、使用日期不匹配、每日时间段不匹配时不能保存。

### 10.3 多模块联动

- 学员详情能看到所有课包和消耗记录。
- 教练运营排课详情能看到课程类型和课包名。
- 教练端看不到价格、实收、支付方式。
- 课程产品被售卖课包引用时不能删除。
- 售卖课包已有购买记录时不能删除，只能停用。
- 非黄金时段课包不能被黄金时段排课消耗；黄金时段课包也不能误用于非黄金时段，除非课包明确配置全天可用。

### 10.4 回归

- 原有学员、班次、学习计划、排课、反馈、教练运营页面不应空白。
- 原有 `npm test` 全部通过。

## 11. 风险和处理

### 风险 1：班次课时和权益账户双轨

处理：
- 第一版保留旧班次/学习计划课时，同时新增权益账户扣课。
- 后续单独做“学习计划余额切换为权益账户汇总”的迁移。

### 风险 2：团课多人扣课复杂

处理：
- 排课记录预留 `entitlementUses`。
- 团课每个学员各自选择自己的权益账户。
- 不允许多个学员共用同一个权益账户，除非以后明确做家庭共享账户。

### 风险 3：历史数据没有权益账户

处理：
- 第一版不强制迁移历史数据。
- 历史班次继续按旧逻辑展示。
- 新购买从权益账户开始。

### 风险 4：管理员选错课包

处理：
- 系统推荐，但必须人工确认。
- 不匹配直接禁止保存。
- 保存后消耗记录可追踪。

### 风险 5：教练端泄露财务数据

处理：
- 后端过滤购买金额、支付方式、经办人。
- 前端教练端不渲染这些字段。

### 风险 6：黄金/非黄金时段误扣

处理：
- 购买时间和使用时间必须分开建模。
- 售卖课包和权益账户都保存 `dailyTimeWindows`，排课保存时由后端校验。
- 一节课必须完整落在某一个可用时间段内；跨出时间段直接禁止保存。
- 推荐课包时显示 `timeBand`，让管理员知道为什么推荐。

### 风险 7：时段规则后续变化影响历史权益

处理：
- 购买后生成权益账户时，把当时课包的时间规则复制到权益账户。
- 后续编辑售卖课包不自动改历史权益账户，避免已经购买的权益被悄悄改变。
- 如果业务要统一调整历史权益，必须做单独的批量调整入口和审计记录。
