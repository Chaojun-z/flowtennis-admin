# FlowTennis 教学履约与权益账户统一开发文档

> 给执行开发者：本文件是后续开发的主依据。实现时必须先读本文档，再读同目录下已有的 `2026-04-11-coach-feedback-course-package-requirements.md`、`2026-04-12-course-package-entitlement-implementation-plan.md` 和 `2026-04-12-membership-management-implementation-plan.md`。  
> 目标是上线后真实投入使用，核心要求是：不产生脏数据、不重复扣课、不漏扣课、不让管理端和教练端看到互相矛盾的信息。

**目标：** 把“课程产品、售卖课包、购买记录、权益账户、教学管理、教练运营、订场会员”统一成一套边界清晰、账本可靠的业务系统。

**核心结论：** 先稳住底层数据和后端校验，再改管理端展示，最后重构教练端。不要先做页面美化。

**技术现状：**

- 后端主文件：`api/index.js`
- 前端主文件：`public/index.html`
- 当前测试：`tests/*.test.js`
- 数据表：Tablestore 运行时表
- 现有主要表：`students`、`products`、`packages`、`purchases`、`entitlements`、`entitlement_ledger`、`classes`、`plans`、`schedule`、`feedbacks`、`coaches`、`campuses`、`courts`
- 会员需求计划新增表：`membership_plans`、`membership_accounts`、`membership_orders`、`membership_benefit_ledger`、`membership_account_events`

## 1. 一句话业务模型

系统必须按下面这条链路理解：

```text
课程产品
  ↓
售卖课包
  ↓
购买记录
  ↓ 自动生成
权益账户
  ↓ 排课时选择/推荐
排课表
  ↓ 自动写入
权益流水
  ↓ 同步影响
权益账户余额 / 班次进度 / 学习计划 / 学员详情 / 教练运营 / 教练端课表
```

各对象职责必须固定：

```text
课程产品 = 课程模板
售卖课包 = 可以卖给学员的商品
购买记录 = 学员买了一次
权益账户 = 学员还剩多少课
权益流水 = 每一次扣课/退课/调整的证据
排课 = 真正发生或即将发生的一节课
反馈 = 这节课上完后的教学记录
```

最关键规则：

```text
课时余额只能由权益流水改变，不能由前端直接改。
```

---

### 1.1 三套账本边界

后续开发必须先分清三套账，不允许混账：

```text
上课课包权益账
  - 主体：学员
  - 用途：排课、上课、消课、课后反馈
  - 主账：entitlements + entitlement_ledger

订场储值资金账
  - 主体：订场用户 courts
  - 用途：订场充值、订场消费、退款、会员充值、会员折扣消费、会员到期清零
  - 主账：courts.history

订场会员权益账
  - 主体：订场用户 courts
  - 用途：会员身份、折扣、赠送服务权益、会员有效期
  - 主账：membership_accounts + membership_orders + membership_benefit_ledger + membership_account_events
```

边界规则：

- 课包权益服务“上课消课”，不能拿来做订场储值。
- 会员系统服务“订场储值、订场折扣、赠送服务权益”，第一版不直接参与排课消课。
- 会员金额余额不新建独立余额账，仍以 `courts.history` 为准。
- 会员赠送权益不进入 `entitlements`，必须独立记在会员权益流水里。
- 学员可以关联订场用户，但会员主账户挂在订场用户，不挂在学员。
- 教练端不显示会员金额、充值记录、支付方式、退款金额。

## 2. 当前薄弱点

### 2.1 多人课程扣课模型不完整

当前排课更偏向：

```text
一条排课 = 一个 entitlementId
```

这只适合单人私教。真实业务会出现：

- 团课
- 多人私教
- 训练营
- 一个班次多个学员

正确模型应该是：

```text
一条排课
  - 学员 A 扣权益账户 A1
  - 学员 B 扣权益账户 B1
  - 学员 C 扣权益账户 C1
```

所以后续必须从单字段 `entitlementId` 升级为结构化明细。

### 2.2 已售出课包仍可能被改规则

售卖课包一旦有购买记录，以下字段不能随便改：

- 课程产品
- 课程类型
- 课时数
- 有效天数
- 可购买日期
- 可使用日期
- 每日可用时间段
- 可用教练
- 可用校区
- 人数限制

否则老学员购买时的规则会被新规则污染。

### 2.3 课时字段分散

当前多个对象都存在课时字段：

- 班次：`totalLessons`、`usedLessons`
- 学习计划：`totalLessons`、`usedLessons`
- 权益账户：`totalLessons`、`usedLessons`、`remainingLessons`

必须明确权威来源：

```text
权益余额：以权益账户 + 权益流水为准
班次进度：以有效排课统计为准
学习计划：由班次、排课、反馈汇总展示
```

不要让多个页面都能独立改课时。

### 2.4 管理端和教练端信息职责不清

管理端要看完整经营信息：

- 产品
- 售卖
- 购买
- 权益
- 扣课
- 排课
- 反馈
- 教练运营

教练端只看教学执行信息：

- 时间
- 校区
- 场地
- 学员
- 班次
- 课程类型
- 剩余课时
- 反馈状态

教练端不能看：

- 课包价格
- 实收金额
- 支付方式
- 购买流水

### 2.5 订场会员容易和课包权益混账

会员需求里有“充值余额、会员折扣、赠送权益、有效期延续、两年清零”。这些规则和上课课包不是一回事。

必须固定：

```text
课包权益 = 学员上课消课
订场会员 = 订场用户充值、折扣、赠送服务
订场余额 = courts.history 资金主账
```

如果把会员赠送权益塞进上课权益账户，会导致：

- 教练端看到不该看的财务或会员信息。
- 订场余额和会员余额变成两套账。
- 赠送权益有效期和课包有效期混在一起。
- 未来退款、清零、到期失效无法追溯。

## 3. 表关系设计

### 3.1 `products` 课程产品

职责：定义课程模板。

建议字段：

```json
{
  "id": "prod-id",
  "name": "成人私教",
  "type": "私教课",
  "maxStudents": 1,
  "price": 0,
  "lessons": 0,
  "notes": "",
  "createdAt": "2026-04-12T00:00:00.000Z",
  "updatedAt": "2026-04-12T00:00:00.000Z"
}
```

规则：

- 可以作为班次和售卖课包的基础模板。
- 不承载活动价、售卖期、使用期、教练限制、校区限制。
- 已被班次或课包使用后不能直接删除。
- 已被使用后，修改 `type`、`maxStudents`、`lessons` 需要谨慎；第一版建议禁止修改这些核心字段。

### 3.2 `packages` 售卖课包

职责：定义能卖给学员的商品规则。

建议字段：

```json
{
  "id": "pkg-id",
  "name": "五一私教 5 节课包",
  "productId": "prod-id",
  "productName": "成人私教",
  "courseType": "私教课",
  "price": 1000,
  "lessons": 5,
  "validDays": 60,
  "saleStartDate": "2026-05-01",
  "saleEndDate": "2026-05-05",
  "usageStartDate": "2026-05-02",
  "usageEndDate": "2026-07-01",
  "dailyTimeWindows": [
    {
      "label": "非黄金时段",
      "startTime": "07:00",
      "endTime": "17:00",
      "daysOfWeek": [1, 2, 3, 4, 5]
    }
  ],
  "timeBand": "非黄金时段",
  "coachIds": ["coach-id"],
  "coachNames": ["朝珺"],
  "campusIds": ["mabao"],
  "maxStudents": 1,
  "status": "active",
  "notes": "",
  "createdAt": "2026-04-12T00:00:00.000Z",
  "updatedAt": "2026-04-12T00:00:00.000Z"
}
```

规则：

- `coachIds` 为空表示不限教练。
- `campusIds` 为空表示不限校区。
- `dailyTimeWindows` 为空表示全天可用。
- `status=inactive` 表示不能继续新购买，不影响历史权益账户。
- 有购买记录后，核心规则不能修改，只允许停用、改备注、改展示名。

### 3.3 `purchases` 购买记录

职责：记录学员的一次购买行为。

建议字段：

```json
{
  "id": "pur-id",
  "studentId": "stu-id",
  "studentName": "张三",
  "studentPhone": "13800000000",
  "packageId": "pkg-id",
  "packageName": "五一私教 5 节课包",
  "productId": "prod-id",
  "productName": "成人私教",
  "courseType": "私教课",
  "packageLessons": 5,
  "packagePrice": 1000,
  "purchaseDate": "2026-05-02",
  "amountPaid": 1000,
  "payMethod": "微信",
  "operator": "管理员",
  "status": "active",
  "notes": "",
  "createdAt": "2026-04-12T00:00:00.000Z",
  "updatedAt": "2026-04-12T00:00:00.000Z"
}
```

规则：

- 新增购买记录时，必须同步创建权益账户。
- 购买记录要保存课包快照，避免后续课包改名或停用影响历史。
- 有消耗流水后，不能改学员、课包、购买日期。
- 不能物理删除有业务关联的购买记录，只能作废。

### 3.4 `entitlements` 权益账户

职责：表示学员可以消耗的课时账户。

建议字段：

```json
{
  "id": "ent-id",
  "studentId": "stu-id",
  "studentName": "张三",
  "purchaseId": "pur-id",
  "packageId": "pkg-id",
  "packageName": "五一私教 5 节课包",
  "productId": "prod-id",
  "courseType": "私教课",
  "totalLessons": 5,
  "usedLessons": 2,
  "remainingLessons": 3,
  "validFrom": "2026-05-02",
  "validUntil": "2026-07-01",
  "usageStartDate": "2026-05-02",
  "usageEndDate": "2026-07-01",
  "dailyTimeWindows": [],
  "timeBand": "全天",
  "coachIds": [],
  "coachNames": [],
  "campusIds": [],
  "maxStudents": 1,
  "status": "active",
  "createdAt": "2026-04-12T00:00:00.000Z",
  "updatedAt": "2026-04-12T00:00:00.000Z"
}
```

规则：

- `remainingLessons = totalLessons - usedLessons`，以后端计算为准。
- `status` 取值：`active`、`expired`、`depleted`、`voided`。
- 第一版不允许前端直接编辑 `usedLessons` 和 `remainingLessons`。
- 调整余额必须写权益流水。

### 3.5 `entitlement_ledger` 权益流水

职责：记录每一次扣课、退课、调整。

建议字段：

```json
{
  "id": "led-id",
  "entitlementId": "ent-id",
  "studentId": "stu-id",
  "scheduleId": "sch-id",
  "lessonDelta": -1,
  "action": "consume",
  "reason": "排课消课",
  "operator": "管理员",
  "createdAt": "2026-04-12T00:00:00.000Z"
}
```

规则：

- 排课扣课：`lessonDelta=-1`。
- 取消排课退课：`lessonDelta=+1`。
- 管理员调整：允许 `lessonDelta` 正负，但必须写明原因。
- 权益流水不允许物理删除。

### 3.6 `schedule` 排课

职责：真实教学履约记录。

当前单人课可继续兼容：

```json
{
  "id": "sch-id",
  "startTime": "2026-05-04 09:00",
  "endTime": "2026-05-04 10:00",
  "classId": "class-id",
  "studentIds": ["stu-id"],
  "studentName": "张三",
  "courseType": "私教课",
  "coach": "朝珺",
  "coachId": "coach-id",
  "campus": "mabao",
  "venue": "1号场",
  "lessonCount": 1,
  "status": "已排课",
  "entitlementId": "ent-id",
  "packageName": "五一私教 5 节课包",
  "notes": ""
}
```

后续多人课必须升级为：

```json
{
  "id": "sch-id",
  "startTime": "2026-05-04 09:00",
  "endTime": "2026-05-04 10:00",
  "classId": "class-id",
  "courseType": "团课",
  "coach": "朝珺",
  "coachId": "coach-id",
  "campus": "mabao",
  "venue": "1号场",
  "lessonCount": 1,
  "status": "已排课",
  "participants": [
    {
      "studentId": "stu-a",
      "studentName": "学员A",
      "entitlementId": "ent-a",
      "packageName": "青少年团课 10 节",
      "lessonCount": 1
    },
    {
      "studentId": "stu-b",
      "studentName": "学员B",
      "entitlementId": "ent-b",
      "packageName": "青少年团课 10 节",
      "lessonCount": 1
    }
  ],
  "notes": ""
}
```

兼容规则：

- 旧数据如果没有 `participants`，读取时由 `studentIds + entitlementId` 临时组装为一个参与明细。
- 新保存的多人排课必须写 `participants`。
- 后续所有扣课、退课、展示都以 `participants` 为准。

```json
  "scheduleId": "sch-id",
  "studentId": "stu-id",
  "studentIds": ["stu-id"],
  "studentName": "张三",
  "coach": "朝珺",
  "startTime": "2026-05-04 09:00",
  "campus": "mabao",
  "venue": "1号场",
  "lessonCount": 1,
  "remainingLessons": 3,
  "practicedToday": "正手稳定性",
  "knowledgePoint": "重心转移",
  "nextTraining": "反手和步伐",
  "sentToStudent": false,
  "createdAt": "2026-04-12T00:00:00.000Z",
```

- 教练只能填写自己的排课反馈。
- 有反馈后，排课的学员、班次、权益、时间、校区、场地、教练、课程类型不能随便改。

### 3.8 订场会员相关表

会员系统和课包权益系统完全分开。会员主挂载对象是订场用户 `courts`，不是学员。

#### 3.8.1 `courts.history` 订场资金主账

职责：继续作为订场充值、消费、退款、会员充值、会员折扣消费、会员到期清零的资金主账。

会员相关流水需增加字段：

```json
{
  "id": "court-his-id",
  "date": "2026-04-12",
  "type": "充值",
  "category": "会员充值",
  "amount": 5000,
  "bonusAmount": 498,
  "membershipOrderId": "mord-id",
  "membershipAccountId": "macc-id",
  "membershipPlanId": "mplan-gold",
  "membershipPlanName": "黄金卡",
  "discountRate": 0.8,
  "originalAmount": 0,
  "discountedAmount": 0,
  "note": "黄金卡开卡"
}
```

订场会员消费需保存原价和折后价：

```json
{
  "type": "消费",
  "category": "会员订场",
  "originalAmount": 300,
  "discountRate": 0.8,
  "discountedAmount": 240,
  "amount": 240,
  "payMethod": "储值扣款",
  "membershipAccountId": "macc-id"
}
```

到期清零流水：

```json
{
  "type": "冲正",
  "category": "会员到期清零",
  "amount": 600,
  "membershipAccountId": "macc-id",
  "note": "两年到期余额清零"
}
```

规则：

- 会员金额余额不能放到 `membership_accounts` 单独计算。
- 余额一律由 `courts.history` 计算。
- 到期清零减少余额，但不能增加累计消费。

职责：标准会员模板，不代表真实成交内容。

```json
{
  "id": "mplan-gold",
  "name": "黄金卡",
  "tierCode": "gold",
  "rechargeAmount": 5000,
  "discountRate": 0.8,
  "bonusAmount": 498,
  "benefitTemplate": {
    "publicLesson": { "label": "大师公开课", "unit": "次", "count": 6 },
    "stringingLabor": { "label": "穿线免手工费", "unit": "次", "count": 5 },
    "ballMachine": { "label": "发球机免费使用", "unit": "次", "count": 6 },
    "partnerHit": { "label": "国家二级运动员陪打", "unit": "次", "count": 2 }
  },
  "validMonths": 12,
```

- 会员方案只是模板。
- 同档位但成交权益不同，差异必须写进会员购买记录快照。
- 已产生会员购买记录的方案不能直接删除，只能停用。

#### 3.8.3 `membership_accounts` 会员账户

职责：订场用户当前会员身份、折扣和有效期主视图。

```json
{
  "id": "macc-id",
  "courtId": "court-id",
  "courtName": "王大人",
  "phone": "15001010368",
  "studentIds": ["stu-id"],
  "status": "active",
  "memberTag": "gold",
  "memberLabel": "黄金卡",
  "discountRate": 0.8,
  "cycleStartDate": "2026-04-05",
  "validUntil": "2027-04-04",
  "hardExpireAt": "2028-04-04",
  "autoExtended": false,
  "lastQualifiedRechargeAmount": 5000,
  "lastOrderId": "mord-id",
  "notes": ""
}
```

规则：

- 一个订场用户最多一个会员账户。
- `status` 取值：`inactive`、`active`、`extended`、`expired`、`cleared`、`voided`。
- 账户余额不放在这里，余额仍从 `courts.history` 算。

```json
  "benefitValidUntil": "2027-04-04",
  "courtHistoryRechargeId": "court-his-id",
  "operator": "管理员",
  "status": "active",
  "notes": ""
}
```

规则：

- 真正成交内容以 `membership_orders` 为准，不以 `membership_plans` 为准。
- 创建会员购买记录时，必须写入 `courts.history` 会员充值流水。
- 开卡/续充可自定义赠送金额和赠送权益次数。

#### 3.8.5 `membership_benefit_ledger` 会员赠送权益流水

职责：记录赠送权益的扣减、补发、作废。

```json
{

  "membershipOrderId": "mord-id",
  "membershipAccountId": "macc-id",
  "courtId": "court-id",
  "benefitCode": "ballMachine",
  "benefitLabel": "发球机免费使用",
  "unit": "次",
  "delta": -1,
  "action": "consume",
  "reason": "会员权益使用",
  "relatedDate": "2026-05-08",
  "operator": "管理员",
  "notes": ""
}
```

规则：

- 每次使用、补发、作废都记流水。
- 必须精确到 `membershipOrderId`，因为赠送权益按购买批次独立失效。
- 赠送权益不进入上课课包权益表。

#### 3.8.6 `membership_account_events` 会员账户事件

职责：记录自动延续、自动清零、有效期重置、退卡作废等账户级事件。

```json
{
  "id": "maevt-id",
  "membershipAccountId": "macc-id",
  "courtId": "court-id",
  "eventType": "auto_extend",
  "beforeStatus": "active",
  "afterStatus": "extended",
  "beforeValidUntil": "2027-04-04",
  "afterValidUntil": "2028-04-04",
```

## 6. 统一后端校验规则

### 6.1 课程产品校验

保存时：

- 名称必填。
- 类型必填。
- 人数限制大于 0。
- 课时不能小于 0。
- 价格不能小于 0。

删除时：

- 已被班次使用，禁止删除。
- 已被售卖课包使用，禁止删除。

编辑时：

- 未被使用：允许完整编辑。
- 已被使用：禁止改核心字段，允许改名称和备注。

### 6.2 售卖课包校验

保存时：

- 课包名称必填。
- 课程产品必须存在。
- 课程类型必填。
- 价格不能小于 0。
- 课时必须大于 0。
- 有效天数不能小于 0。
- 人数限制必须大于 0。
- 活动结束日期不能早于活动开始日期。
- 可用结束日期不能早于可用开始日期。
- 每日可用时段必须成对填写。
- 每日可用时段结束时间必须晚于开始时间。
- 可用教练必须存在。
- 可用校区必须存在。

编辑时：

- 没有购买记录：允许完整编辑。
- 已有购买记录：禁止修改核心规则，只能停用、改名称、改备注。

删除时：

- 没有购买记录：允许删除。
- 已有购买记录：禁止删除，只能停用。

### 6.3 购买记录校验

新增时：

- 学员必须存在。
- 售卖课包必须存在。
- 售卖课包必须启用。
- 购买日期必须在可购买日期范围内。
- 实收金额不能小于 0。
- 支付方式必填。
- 保存购买记录成功后，必须创建权益账户。
- 如果创建权益账户失败，必须回滚购买记录。

编辑时：

- 没有消耗流水：允许改学员、课包、购买日期、实收金额、支付方式、备注。
- 已有消耗流水：禁止改学员、课包、购买日期，只允许改备注。

作废时：

- 没有消耗流水：购买记录和对应权益账户可标记为作废。
- 已有消耗流水：禁止直接作废。

### 6.4 权益账户校验

保存时：

- 必须来源于购买记录。
- 不允许前端直接改 `usedLessons`。
- 不允许前端直接改 `remainingLessons`。
- 余额变化必须通过权益流水。

消费时：

- 状态必须为 `active`。
- 剩余课时必须足够。
- 学员必须匹配。
- 课程类型必须匹配。
- 教练必须匹配。
- 校区必须匹配。
- 上课日期必须在可用日期范围内。
- 上课时间必须在每日可用时间段内。
- 人数限制必须匹配。

### 6.5 排课校验

保存时：

- 开始时间必填。
- 结束时间必填。
- 结束时间必须晚于开始时间。
- 教练必填且必须存在。
- 校区必填且必须存在。
- 场地必填。
- 至少一个学员。
- 学员必须存在。
- 如果关联班次，班次必须存在。
- 如果关联班次，排课学员必须属于该班次。
- 如果关联班次，教练应与班次教练一致；如允许临时代课，必须显式记录代课原因。
- 教练同时间不能冲突。
- 学员同时间不能冲突。
- 同校区同场地同时间不能冲突。
- 与订场用户的场地占用不能冲突。
- 跨校区间隔小于 60 分钟要提醒。
- 每个学员必须绑定一个可用权益账户。
- 每个权益账户必须通过权益消费校验。

编辑时：

- 没有反馈、没有权益流水：允许按保存规则编辑。
- 已有权益流水：改学员、权益、课时数、状态时必须先做退课再重新扣课。
- 已有反馈：禁止改学员、班次、权益、时间、校区、场地、教练、课程类型。

取消时：

- 状态改为 `已取消`。
- 已扣课的每个参与学员都要写退课流水。
- 不物理删除已产生权益流水的排课。

删除时：

- 没有反馈、没有权益流水：允许删除。
- 有反馈或权益流水：禁止删除，只能取消。

### 6.6 课后反馈校验

新增时：

- 排课必须存在。
- 当前用户必须是管理员或该排课教练。
- `今天练习了` 必填。
- `下节课我们训练` 必填。

编辑时：

- 当前用户必须是管理员或该排课教练。
- 反馈仍然绑定原排课。

### 6.7 订场会员校验

会员方案保存时：

- 名称必填。
- 档位代码必填。
- 充值金额必须大于 0。
- 折扣必须大于 0 且小于等于 1。
- 赠送金额不能小于 0。
- 赠送权益次数不能小于 0。
- 有效月数必须大于 0。
- 最长有效月数必须大于等于有效月数。

会员方案删除时：

- 没有会员购买记录：允许删除。
- 已有会员购买记录：禁止删除，只能停用。

开卡 / 续充时：

- 订场用户必须存在。
- 会员方案必须存在且启用。
- 充值金额必须大于 0。
- 购买日期必填。
- 如果没有会员账户，创建会员账户。
- 创建会员购买记录。
- 向 `courts.history` 写会员充值流水。
- 固化赠送权益快照。
- 更新会员账户状态、折扣、有效期。

有效期内续充时：

- 如果续充金额 `>= lastQualifiedRechargeAmount`，重置当前余额有效期。
- `cycleStartDate = 本次续充日`。
- `validUntil = 本次续充日 + 12 个月`。
- `hardExpireAt = 本次续充日 + 24 个月`。
- 当前余额不清空。
- 当前折扣更新为本次档位折扣。
- 新生成一条会员购买记录和新赠送权益批次。

低于原档位续充时：

- 不重置会员有效期。
- 第一版建议作为普通储值处理。
- 系统必须明确提示管理员。

一年到期自动延续：

- 当 `today > validUntil`、`today <= hardExpireAt`、当前余额 `> 0`、且尚未自动延续时，状态切为 `extended`。
- 写 `membership_account_events`。
- 不新增会员购买记录。
- 折扣规则不变。

两年到期自动清零：

- 当 `today > hardExpireAt` 且当前余额 `> 0` 时，写 `courts.history` 清零流水。
- 清零流水减少余额，但不能增加累计消费。
- 会员账户状态改为 `cleared`。
- 写 `membership_account_events`。

赠送权益：

- 每次开卡/续充生成独立赠送权益批次。
- 权益有效期按 `membership_orders.benefitValidUntil`。
- 使用权益必须指定购买批次。
- 到期未使用自动失效，不延期、不折现。
- 退卡或作废时，未使用赠送权益随之作废。

### 6.8 订场财务主账校验

`courts.history` 必须继续作为订场资金主账。

会员充值：

- 增加余额。
- 增加累计充值。
- 增加累计实收。
- 保存会员订单、会员账户、会员方案快照字段。

会员订场消费：

- 保存原价 `originalAmount`。
- 保存折扣 `discountRate`。
- 保存折后价 `discountedAmount`。
- 余额扣减按折后价。
- 累计消费按折后价。

会员到期清零：

- 减少余额。
- 不增加累计消费。
- 不增加累计实收。
- 必须关联 `membershipAccountId`。

权限：

- 管理员可见会员金额、购买记录、赠送权益、作废记录。
- 教练端 `load-all` 默认不返回会员相关数组。
- 学员详情只能展示关联订场账户的会员摘要。

## 9. 上线验收标准

上线前必须满足：

- 新购买一定生成权益账户。
- 新排课一定绑定权益账户或参与明细。
- 单人课不会漏扣。
- 多人课不会只扣一个人。
- 取消排课能退回对应课时。
- 权益流水能追溯每一次扣课和退课。
- 管理端学员详情能看到购买、权益、排课、反馈。

- 两年到期清零不增加累计消费。
- 会员赠送权益能按购买批次追溯。
- 学员详情能看到关联订场账户会员摘要。
- 教练端不显示财务敏感信息。
- 教练端不显示会员金额、会员充值记录、会员退款记录。
- 教练端能看到完整上课信息。
- 删除核心数据时不会破坏历史链路。
- 数据体检无严重错误。
- `npm test` 通过。

## 10. 建议执行顺序

严格按下面顺序做：

```text
0. 全局数据边界、权限过滤、字段冻结
1. 上课课包权益后端规则和测试
2. 多学员权益扣课模型
3. 管理端排课弹窗和详情
4. 管理端学员/班次/权益详情串联
5. 订场会员后端模型和 courts.history 财务主账扩展
6. 订场会员管理端页面和订场用户/学员联动
7. 教练运营增强
8. 教练端工作台重构
9. 数据体检和上线验收
```

不要先做教练端页面。  
原因：管理端是数据源，教练端只是使用视角。管理端不稳，教练端一定会乱。

会员系统可以在阶段 1 完成后并行进入阶段 5，但不能早于阶段 0。  
原因：会员不直接阻塞排课消课，但它涉及订场资金主账，必须先保证权限过滤和账本边界清楚。

## 11. 开发纪律

- 每个阶段独立提交。
- 每个阶段先写测试，再改实现。
- 不做大重构。
- 不改无关页面。
- 不把核心业务信息继续塞进备注。
- 后端校验必须比前端校验更严格。
- 前端只负责提示，不能作为唯一校验。
- 所有敏感信息不能硬编码。
- 出现连续两次同类报错，先停下来定位原因，再继续。
