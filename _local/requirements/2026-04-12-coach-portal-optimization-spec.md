# FlowTennis 教练端优化方案

> 本文档用于指导教练端的页面结构、信息层级、操作流程和体验课转化记录设计。  
> 本文档重点不是底层数据建模，而是从教练真实工作流出发，定义“教练端应该怎么组织信息、怎么减少漏事、怎么让运营接得住体验课转化”。  
> 后续开发教练端时，必须先读本文档，再读：
> - `_local/requirements/2026-04-12-teaching-entitlement-unified-development-plan.md`
> - `_local/requirements/2026-04-11-coach-feedback-course-package-requirements.md`

---

## 1. 文档目标

当前教练端的核心问题不是视觉样式，而是：

1. 信息层级混乱，教练找课、找学员、找待反馈都慢。
2. 一节课的信息不完整，时间、校区、场地、学员、班次、备注没有稳定聚合。
3. 页面是“数据页面”，不是“工作台页面”。
4. 体验课反馈只记录教学内容，不足以支撑后续运营转化跟进。
5. 管理端和教练端之间缺少一条清晰的“体验课观察 -> 运营接手”的链路。

本次优化目标：

- 让教练进入系统后，先看到最重要的事。
- 让教练在一处看完整一节课需要的信息。
- 让课后反馈真正融入工作流，而不是课后补填的负担。
- 让体验课信息既服务教学，又服务转化和后续运营跟进。

---

## 2. 教练端的产品定位

教练端不是管理后台的缩小版，也不是一个数据查询页。

教练端本质上是：

```text
教练的每日工作台
```

它应该围绕下面这条主线设计：

```text
看今天要上什么课
  ↓
快速理解每节课的背景和关键信息
  ↓
完成上课
  ↓
立刻做课后反馈 / 体验课判断
  ↓
把需要运营接手的线索交出去
```

所以教练端最重要的不是“数据齐全”，而是：

- 先看最重要的
- 先做最紧急的
- 不漏反馈
- 不漏转化线索

## 3. 我们对教练真实需求的理解

## 3.1 教练每天最关心的事

教练进入系统后，优先级通常是：

1. 今天我有几节课？
2. 下一节课几点、在哪个校区、哪个场地、带谁上？
3. 这节课是正式课还是体验课？
4. 这个学员/这组学员我上课前要注意什么？
5. 哪些课上完后还没反馈？
6. 哪些体验课需要我给出转化判断，让运营跟进？

因此教练端首页不应该先让教练看“大表格”，而应该先给出：

- 今日任务
- 即将开始
- 待反馈
- 体验课待判断

## 3.2 一节课对教练来说最少必须看到什么

一节课的基础信息必须稳定包含：

- 开始时间
- 结束时间
- 校区
- 场地
- 课程类型
- 班次名称
- 学员姓名 / 学员列表
- 是否体验课
- 当前状态（待上课 / 已完成 / 已取消 / 待反馈）
- 课程备注

如果是体验课，还必须额外看到：

- 学员当前基础水平
- 过往打球/上课经验
- 来体验的目标
- 当前主要问题
- 转化潜力
- 更适合推荐什么产品
- 是否需要运营继续跟进

结论：

```text
正式课看教学执行信息
体验课还要看转化判断信息
```

## 4. 当前教练端的问题拆解

## 4.1 首页信息没有按优先级排列

现在“我的课表 / 我的学员 / 我的班次”是功能目录视角，不是工作流视角。

这会导致：

- 教练先自己判断该点哪个页面
- 今天最急的任务没有在首页被高亮
- 待反馈课程和即将开始课程没有被优先处理

## 4.2 一节课的信息分散

时间、场地、学员、校区、班次、课程类型、备注没有稳定聚合成一个“课程信息单元”。

结果就是：

- 教练要自己拼信息
- 体验课前无法快速准备
- 反馈时上下文不完整

## 4.3 体验课只被当成“特殊课程”，没有被当成“转化线索”

这是一个明显缺口。

现实里体验课结束后，教练通常会判断：

- 这个学员是零基础还是有基础
- 是偏爱订场，还是更适合课程产品
- 是更适合私教、训练营、班课，还是先继续观察
- 成交意愿高不高
- 是否值得运营立即跟进

这些判断如果不被结构化记录：

- 运营只能靠聊天和口头沟通
- 后续跟进容易断层
- 教练每次都得重复解释

## 4.4 教练端和管理端之间没有完整交接链路

当前问题不是“教练记了，运营没看”，而是系统里缺少明确的交接对象。

正确链路应该是：

```text
体验课完成
  ↓
教练填写体验课反馈 + 转化判断
  ↓
管理端生成可筛选的待跟进线索
  ↓
运营接手
```

## 5. 教练端优化后的整体结构

## 5.1 导航结构建议

教练端保留现有 3 个主模块，但首页和内容重心要重排。

建议结构：

1. `工作台`
2. `我的课表`
3. `我的学员`
4. `我的班次`

说明：

- `工作台` 是新的教练首页，承接高频工作。
- `我的课表` 是完整排课视图。
- `我的学员` 是教学关系视图。
- `我的班次` 是班课/长期班次视图。

## 5.2 为什么要新增“工作台”

因为教练进入系统的第一需求，不是查所有数据，而是：

- 我今天先做什么
- 我马上要上什么
- 我还有什么没反馈
- 哪些体验课需要我补判断

如果没有 `工作台`，教练每天都要先自己做一次“任务拆解”。

## 6. 工作台设计

## 6.1 顶部摘要区

工作台顶部显示 4 个摘要卡：

1. 今日课程数
2. 即将开始课程数（例如未来 2 小时）
3. 待反馈课程数
4. 体验课待判断数

这 4 个数字必须可点击，点击后直接筛到对应列表。

## 6.2 主任务区块排序

工作台主区块按下面顺序排列：

### A. 即将开始

显示最近要上的 3-5 节课。

每条课程卡最少显示：

- 时间
- 校区
- 场地
- 学员
- 课程类型
- 是否体验课
- 进入详情按钮

### B. 今日全部课程

按时间顺序排全天课程。

### C. 待反馈

显示今天或近期上完但还没反馈的课程。

每条记录必须有明显按钮：

- `填写反馈`

### D. 体验课待判断

显示已上完但“转化判断未完成”的体验课。

每条记录显示：

- 学员
- 时间
- 校区
- 课程教练
- 当前反馈状态
- `补充转化判断`

这里必须和普通“待反馈”区分开。

## 7. 我的课表页怎么改

## 7.1 课表页核心目标

让教练快速看清：

- 我什么时候有课
- 这节课到底带谁
- 这节课是不是体验课
- 上完课要不要立刻反馈

## 7.2 课表卡片 / 课表块字段

每一节课在课表视图里至少展示：

- 开始时间 - 结束时间
- 校区
- 场地
- 学员姓名 / 多名学员摘要
- 课程类型
- 班次名称
- 标签：体验课 / 正式课 / 私教 / 团课
- 标签：待反馈 / 已反馈

## 7.3 点击课程后的详情抽屉

点击课表中的一节课，右侧打开课程详情抽屉，而不是让教练跳多个页面。

课程详情抽屉必须分 4 块：

### 1. 课程基础信息

- 时间
- 校区
- 场地
- 课程类型
- 班次
- 学员

### 2. 上课前信息

- 学员备注
- 历史问题
- 教练备注
- 本节课关注点

### 3. 课后动作

- 填写反馈
- 查看历史反馈

### 4. 如果是体验课，增加“转化判断”

- 水平判断
- 当前目标
- 适合推荐的产品
- 成交意愿
- 是否需要运营跟进

---

## 8. 我的学员页怎么改

## 8.1 学员页不是通讯录，而是教学对象管理页

教练进入“我的学员”时，不是只想知道姓名和手机号，而是想知道：

- 我最近在带谁
- 这个学员现在上什么班 / 什么课
- 最近一次上课是什么时候
- 最近有哪些问题
- 有没有体验课线索需要继续跟进

## 8.2 学员列表建议字段

- 学员姓名
- 类型（正式 / 体验 / 班课 / 私教）
- 当前班次
- 最近上课
- 最近反馈摘要
- 特殊备注

## 8.3 学员详情建议模块

学员详情至少包含：

1. 基础信息
2. 当前班次 / 当前课程关系
3. 最近上课记录
4. 最近反馈摘要
5. 如果是体验课学员，显示转化建议摘要

如果学员关联了订场用户，还可以只读显示：

- 关联订场账户
- 是否会员

但不要在教练端展示充值金额、支付信息。

---

## 9. 我的班次页怎么改

## 9.1 班次页目标

班次页的重点不是展示所有字段，而是让教练知道：

- 我当前在带哪些班
- 每个班有哪些学员
- 这个班上到哪了
- 下一节课是什么时候

## 9.2 班次列表建议字段

- 班次名称
- 课程产品
- 学员人数
- 学员名单摘要
- 课时进度
- 最近一次上课
- 下一次排课
- 状态

班次详情中应能看到：

- 班级成员
- 近期排课
- 近期反馈

## 10. 体验课的独立设计

体验课不能只是普通课程多一个标签，而必须作为一类特殊任务。

## 10.1 体验课的目标不止教学

体验课结束后，教练需要输出两类判断：

### A. 教学判断

- 学员基础
- 技术问题
- 本节课表现
- 适合什么训练方式

### B. 转化判断

- 转化意愿高 / 中 / 低
- 更适合推什么
- 需不需要运营马上跟进
- 应该怎么跟进

## 10.2 体验课反馈应新增的结构化字段

建议至少包括：

- `playerLevel`
  - 零基础 / 初学 / 有基础 / 长期打球

- `goalType`
  - 健身 / 入门 / 提升技术 / 社交 / 比赛 / 陪练

- `experienceBackground`
  - 无经验 / 有少量体验课 / 打球半年内 / 长期打球

- `mainIssues`
  - 主要问题，文本

- `conversionIntent`
  - 高 / 中 / 低

- `recommendedProductType`
  - 场地会员 / 私教 / 训练营 / 班课 / 继续观察

- `recommendedReason`
  - 推荐原因，文本

- `needOpsFollowUp`
  - 是 / 否

- `opsFollowUpPriority`
  - 高 / 中 / 低

- `opsFollowUpSuggestion`
  - 运营如何跟进，文本

## 10.3 体验课转化判断示例

例如：

### 情况 1

学员平时会自己约球，有一定水平，更多是稳定打球需求。

教练应记录：

- 有基础
- 平时有订场习惯
- 更适合推荐场地会员
- 转化意愿中高
- 建议运营按场地会员方向跟进

### 情况 2

学员零基础，想系统学球。

教练应记录：

- 零基础
- 更适合训练营或私教
- 需要先建立训练关系
- 不建议直接推场地会员

## 10.4 这条信息必须给管理端看到

体验课反馈不能只存在教练端的文本备注里。

必须实现：

```text
教练填写体验课转化判断
  ↓
管理端可筛选查看
  ↓
运营可据此跟进
```

所以管理端后续最少要能筛：

- 待跟进体验课
- 高意愿体验课
- 推荐场地会员的体验课
- 推荐私教/训练营的体验课

## 11. 教练端必须依赖的最小数据字段

为了实现上面的教练体验，系统必须至少稳定提供以下数据。

## 11.1 排课级字段

- `id`
- `startTime`
- `endTime`
- `campus`
- `venue`
- `courseType`
- `classId`
- `className`
- `studentIds`
- `studentNames`
- `coach`
- `status`
- `isTrial`
- `notes`
- `feedbackStatus`

## 11.2 学员摘要字段

- `studentId`
- `studentName`
- `studentType`
- `recentClassName`
- `lastLessonDate`
- `recentFeedbackSummary`
- `specialNotes`

## 11.3 体验课反馈字段

- `playerLevel`
- `goalType`
- `experienceBackground`
- `mainIssues`
- `conversionIntent`
- `recommendedProductType`
- `recommendedReason`

- `opsFollowUpPriority`
- `opsFollowUpSuggestion`

这些字段不一定都在教练端自己建表，但必须让教练端能稳定读写。

## 12. 教练端绝对不该看到的信息

教练端信息边界必须明确：

不能显示：

- 课包价格
- 购买金额
- 支付方式
- 会员充值记录
- 退款金额
- 财务流水细节

可以显示但应控制粒度：

- 是否会员（只读标签）
- 是否有可用上课权益
- 是否体验课

原则：

```text
教练端只看教学执行和转化判断所必需的信息
不看经营财务明细
```

## 13. 页面文案和动作命名建议

为了让教练理解成本更低，按钮名称要更直接。

建议：

- `填写反馈`
- `补充转化判断`
- `查看课程详情`
- `查看学员情况`
- `查看班次详情`

不要使用过于管理后台化的词：

- 编辑记录
- 维护数据
- 新增条目

## 14. 开发优先级

## P0 必做

1. 新增教练端 `工作台`
2. 首页增加：今日课程 / 即将开始 / 待反馈 / 体验课待判断
3. 课表点击后打开课程详情抽屉
4. 课程详情中补齐时间、校区、场地、学员、班次、课程类型
5. 体验课反馈增加结构化转化字段
6. 管理端能看到体验课转化判断结果

## P1 应做

1. 我的学员页重构成教学对象视图
2. 我的班次页增加进度和下一次排课信息
3. 体验课线索在管理端支持筛选和状态跟进

## P2 后做

1. 教练端移动端适配
2. 体验课跟进提醒
3. 教练个人待办通知

## 15. 移动端兼容原则

本期先按网页教练端设计，但要提前考虑未来移动端。

原则：

1. 最重要的信息必须在一屏内能看到：
   - 时间
   - 校区
   - 场地
   - 学员
   - 是否体验课
   - 是否待反馈

2. 体验课转化判断字段不能过深嵌套，避免未来移动端难填。

3. 课程详情抽屉未来可以演化成移动端详情页，但字段结构不变。

---

## 16. 验收标准

教练端优化完成后，至少满足：

1. 教练一进入系统，先看到今天最重要的事。
2. 教练不用切多个页面，就能看完整一节课的关键信息。
3. 体验课和正式课在信息结构上有明显区别。
4. 教练能快速完成课后反馈。
5. 教练能记录体验课转化建议。
6. 管理端能看到教练给出的体验课转化判断，并用于后续跟进。
7. 教练端仍然不暴露财务和购买敏感信息。

---

## 17. 一句话总结

教练端优化不是“把现有表格变好看”，而是：

```text
把教练端从数据页面，改成一个围绕上课、反馈、转化判断展开的工作台
```
