# 排课表页面 90 分改造方案与执行计划

## 1. 文档目的

本文档用于指导「排课表页面」从当前“能记录和管理课程”的状态，升级为更符合真实网球俱乐部管理员、教务、课程顾问、运营人员日常使用的“排课运营中枢”。

本次改造坚持 5 个原则：

1. 先修正当前逻辑问题和数据风险，再补运营字段。
2. 信息层级要清楚：列表负责快速判断，详情负责追溯理解，弹窗负责新增/编辑。
3. 严格控制联动边界：排课页管理单次上课，不把班次、课包、学员、订场、反馈页面的职责全部塞进来。
4. 不做复杂功能：不做自动通知系统、不做审批流、不做复杂多学员权益模型、不重构成完整日历系统。
5. 改造目标是门店每天能用，而不是做看起来复杂的后台。

---

## 1.1 2026-04-13 执行前最新代码对齐记录

在正式执行排课表改造前，已对齐其他线程对「学员信息」「班次管理」「学习计划」的最新改动。

已确认以下内容已经由其他线程完成或部分完成：

- 排课弹窗已使用学员库搜索选择，不再把学员自由文本作为正式关联。
- 班次管理已把 `已上课时` 改为由排课自动维护，且有排课后已保护课程产品、教练、校区、学员、总课时、已上课时等核心字段。
- 学习计划已收口为由班次自动生成，不允许独立新增、修改或删除。
- 学习计划页已经有「去排课」入口，并已强调真实可约课以权益账户规则为准。
- 删除排课的后端保护已存在：已有课后反馈或权益消耗流水的排课不能直接删除。

因此，本轮排课表实施时做以下调整：

- `scheduleSource` 不再写成“后续接入学习计划入口”，而是当前就覆盖 `排课表`、`教练运营`、`学习计划` 三个入口。
- 已反馈排课锁定 `studentName` 只限制“排课编辑接口修改课程事实”，不阻止“学员资料页的身份信息同步”。
- 删除保护不从零实现，重点改为补测试、前端主入口改成取消排课、保留误建删除边界。
- 第一批实施顺序调整为：有效状态口径、取消/删除边界、多人班次提前提示、已反馈锁核心字段、扣课状态与无权益账户确认。

## 2. 当前页面定位

当前排课表已经具备基础能力：

- 搜索排课
- 按状态筛选
- 按校区筛选
- 按正式课 / 体验课筛选
- 分页查看
- 新增排课
- 编辑排课
- 查看排课详情
- 删除排课
- 填写 / 编辑课后反馈
- 关联学员
- 关联班次
- 关联教练
- 关联校区和场地
- 关联权益账户
- 保存时联动班次 / 学习计划课时
- 保存时联动权益账户和权益流水
- 保存时检查教练、学员、场地、订场冲突
- 在教练运营、教练端工作台、我的课表、学员页中被引用

当前真实定位更接近：

`单次课程记录页`

而不是：

`排课运营管理页`

也就是说，它已经能“把课排进去”，但还不够稳定地支撑“每天安排课、确认课、取消课、扣课、反馈、追责和复盘”。

## 3. 当前主要问题与风险

以下问题列为第一优先级，必须先改。

### P0-1. “已结束”状态筛选口径不一致

当前页面会在列表渲染时，把已经过了下课时间的 `已排课` 临时显示成 `已结束`。

但筛选 `已结束` 时，逻辑仍先按原始 `status` 字段筛选，再做临时状态转换。

风险：

- 列表里看起来是“已结束”的课，筛选“已结束”时可能搜不到。
- 运营想找“已结束但未反馈”的课，会漏数据。
- 教练端待反馈、教练运营统计和排课表筛选口径容易不一致。

目标：

- 增加统一的“有效排课状态”计算口径。
- 列表展示、状态筛选、待反馈判断尽量使用同一口径。
- 不强制第一版把状态自动写回数据库，避免引入额外批处理复杂度。

建议口径：

- `已取消` 永远显示为 `已取消`
- 原始状态为 `已结束` 显示为 `已结束`
- 原始状态为 `已排课` 且 `endTime < 当前时间`，展示和筛选按 `已结束`
- 原始状态为 `已排课` 且未过下课时间，展示和筛选按 `已排课`

### P0-2. 删除和取消的边界不清楚

当前列表和弹窗里直接显示“删除”。

后端已有保护：

- 已有课后反馈不能直接删除
- 已有权益消耗流水不能直接删除
- 有权益消耗时提示应该取消排课

风险：

- 对运营来说，“删除”和“取消”不是一回事。
- 删除代表这条记录从业务里消失；取消代表这节课没有上，但仍保留安排、扣课回滚和原因。
- 真实门店里，取消课非常常见，直接显示删除容易引导误操作。

目标：

- 列表主操作从“删除”改为“取消排课”。
- 删除只作为低频操作，用于误建且没有反馈、没有权益流水的排课。
- 取消排课必须保留记录，并触发已有的课时 / 权益退回逻辑。

第一版边界：

- 已扣权益但无反馈：只能取消，不能删除。
- 已有反馈：不能删除，也不能取消或修改课程事实，只允许改备注 / 反馈相关内容。
- 未扣权益、无反馈、无权益流水：允许删除，但放在二级入口或确认文案中明确是“误建删除”。

### P0-3. 选择多人班次后，保存时又禁止多人排课

当前选择班次后，会自动带出班次下所有学员。

但保存时，如果学员数大于 1 且不是取消状态，会提示：

`多人排课暂不支持单个权益账户，请拆成单人排课`

风险：

- 运营选择多人班次后，以为可以保存，最后才被拦截。
- 对班课 / 训练营场景不友好。
- 当前系统还没做“多人排课逐人扣课”，前端应该提前讲清楚限制。

目标：

- 在选择多人班次时立即提示。
- 如果当前阶段不做多人权益模型，就不要让运营误以为能一次保存。

第一版建议：

- 选择多人班次后，弹窗提示：`当前班次包含多名学员，当前版本请拆成单人排课，避免扣课错误。`
- 保存前继续保留后端拦截，防止绕过前端。
- 不在本期做多人排课逐人扣课模型。

### P0-4. 权益账户可以为空，扣课状态表达不清楚

当前保存时允许不绑定课包权益账户。

实际影响：

- 如果关联班次，会扣班次 / 学习计划课时。
- 如果没有权益账户，不会扣权益账户，也不会写权益流水。

风险：

- 运营可能以为“排课成功 = 课包已扣课”，但实际权益账户没有扣。
- 学员页、学习计划页、权益账户页对“还剩多少课”的判断可能不一致。
- 后续续费、到期、欠课排查会变难。

目标：

- 不强制第一版必须绑定权益账户，但必须让运营看懂扣课结果。
- 增加“扣课状态”表达。

建议字段：

- `lessonChargeStatus`
  - `已扣课`
  - `未扣课`
  - `不扣课`
  - `扣课异常`

第一版计算规则：

- 状态为 `已取消`：显示 `不扣课`
- 有 `entitlementId` 且已有对应权益流水：显示 `已扣课`
- 有 `entitlementId` 但未找到流水：显示 `扣课异常`
- 无 `entitlementId` 但 `lessonCount > 0`：显示 `未扣课`

说明：

- 第一版可以先做前端计算展示，不一定新增强制后端字段。
- 但保存时如果权益账户为空，必须弹出确认提示：`本次不会扣课包权益，只会保存排课记录，并可能联动班次课时。`

### P0-5. 已反馈排课的核心字段保护不够完整

当前已有课后反馈后，只禁止修改：

- 学员
- 班次
- 权益账户

但仍可能修改：

- 上课时间
- 下课时间
- 教练
- 校区
- 场地
- 课程类型
- 课程性质
- 消课节数
- 状态

风险：

- 课后反馈代表这节课已经发生或至少已经进入教学记录。
- 如果反馈后还能改时间、教练、场地、课时数，反馈和真实课程事实会不一致。
- 教练工作量、权益扣课、学员历史、运营复盘都会被污染。

目标：

- 已有反馈后，排课核心字段进入保护状态。
- 只允许修改低风险字段。

建议保护字段：

- `studentIds`
- `studentName`
- `classId`
- `entitlementId`
- `startTime`
- `endTime`
- `coach`
- `coachId`
- `campus`
- `venue`
- `courseType`
- `isTrial`
- `lessonCount`
- `status`

允许修改字段：

- `notes`
- 后续如果需要，可允许 `notifyStatus`，但不建议影响课程事实。

### P0-6. 保存前没有确认摘要

当前点击保存后直接提交。

风险：

- 运营排课时最容易错的是：时间、校区、场地、教练、学员、扣哪个课包。
- 这些错误一旦保存后联动扣课，就会牵连班次、权益流水、反馈、教练工作量。

目标：

- 保存前增加轻量确认摘要。
- 不做复杂审核流，只做一次确认。

确认摘要建议展示：

- 学员
- 上课 / 下课时间
- 教练
- 校区 / 场地
- 班次
- 课程类型 / 正式或体验
- 消课节数
- 权益账户 / 扣课状态
- 如果权益账户为空，突出提示“本次不会扣课包权益”

### P0-7. 缺少取消原因

当前只有 `status=已取消`，没有取消原因。

风险：

- 无法区分学员请假、教练请假、天气/场地、运营调整。
- 后续无法判断取消课主要是学员问题、教练问题还是场地问题。
- 对真实俱乐部复盘和服务沟通不够。

目标：

- 增加 `cancelReason` 字段。
- 当状态为 `已取消` 时显示和保存取消原因。

建议选项：

- 学员请假
- 教练请假
- 天气 / 场地
- 临时调整
- 体验课未到
- 其他

第一版不做复杂责任归因，不做罚课规则。

### P0-8. 缺少通知 / 确认状态

当前排课记录没有表达“是否通知到学员和教练”。

风险：

- 门店实际运营最怕“课排了，但学员或教练不知道”。
- 现在系统只能看到课存在，看不到沟通动作是否完成。

目标：

- 增加非常轻量的通知状态，不接入自动消息系统。

建议字段：

- `notifyStatus`
  - `未通知`
  - `已通知学员`
  - `已通知教练`
  - `都已通知`

可选字段：

- `confirmStatus`
  - `待确认`
  - `已确认`

第一版建议：

- 优先做 `notifyStatus`。
- `confirmStatus` 如果页面空间允许就一起做；如果要控制范围，可以先不做。

### P0-9. 排课来源缺失，但第一版优先级低于通知和取消原因

排课可能来自：

- 排课表手动新增
- 教练运营页点击格子创建
- 班次入口进入排课
- 学员 / 学习计划入口进入排课

价值：

- 排查课从哪里来的。
- 判断哪些入口真正被运营使用。

目标：

- 第一版可以先作为隐藏字段自动写入。
- 不要求列表展示。

建议字段：

- `scheduleSource`
  - `排课表`
  - `教练运营`
  - `班次`
  - `学员`
  - `学习计划`

## 4. 从网球俱乐部管理员 / 运营视角，页面应该解决什么问题

真实俱乐部管理员、教务、运营打开排课表，不是为了看一堆课程记录，而是要快速回答下面问题。

### 4.1 今天哪些课要上

需要快速判断：

- 时间
- 学员
- 教练
- 校区
- 场地
- 课程类型
- 是否体验课
- 是否已通知
- 是否已确认

### 4.2 这节课能不能排

需要系统帮忙校验：

- 教练是否冲突
- 学员是否冲突
- 场地是否冲突
- 订场用户是否已占场
- 班次课时是否够
- 课包权益是否可用
- 课包是否匹配课程类型、教练、校区、时间段

当前已有大部分后端校验，页面需要把“为什么不能排 / 保存后会怎么扣课”说清楚。

### 4.3 这节课有没有扣对课

需要快速判断：

- 扣的是哪个权益账户
- 扣了几节课
- 是否没扣课
- 是否因为体验课或特殊原因不扣课
- 取消后是否退回

当前权益流水已经存在，但页面表达不够清楚。

### 4.4 这节课结束后有没有反馈

需要快速判断：

- 是否待反馈
- 是否已反馈
- 体验课是否有转化判断
- 是否需要运营跟进

当前已有反馈入口，但“已结束筛选”和“待反馈口径”需要统一。

### 4.5 这节课如果取消，为什么取消

需要快速判断：

- 是学员原因
- 是教练原因
- 是场地 / 天气原因
- 是运营临时调整

这对门店排课质量和服务复盘非常重要。

## 5. 信息层级设计

### 5.1 列表层：只放运营每天判断需要的信息

列表建议展示：

- 上课时间
- 下课时间
- 学员
- 教练
- 校区 / 场地
- 班次
- 课程标签：正式课 / 体验课、课程类型
- 状态：已排课 / 已结束 / 已取消
- 通知状态
- 扣课状态
- 反馈状态
- 操作：详情、编辑、取消排课

不建议列表展示：

- 完整权益流水
- 学员全部历史备注
- 购买记录明细
- 复杂转化分析

原因：

- 列表的职责是快速判断“今天要处理哪节课”。
- 复杂内容放详情页，不要挤在表格里。

### 5.2 详情层：负责追溯一节课

详情页建议分区：

1. 课程事实
   - 时间
   - 学员
   - 教练
   - 校区 / 场地
   - 班次
   - 课程类型
   - 正式 / 体验

2. 运营状态
   - 通知状态
   - 确认状态
   - 排课来源
   - 状态
   - 取消原因

3. 扣课信息
   - 权益账户
   - 消课节数
   - 扣课状态
   - 相关权益流水摘要

4. 上课前信息
   - 学员备注
   - 历史问题
   - 教练备注

5. 课后信息
   - 反馈状态
   - 反馈摘要
   - 体验课转化判断
   - 运营跟进建议

### 5.3 新增 / 编辑弹窗：只负责本节课可编辑字段

弹窗建议字段：

- 上课时间
- 下课时间
- 关联班次
- 学员
- 课程类型
- 课程性质
- 教练
- 校区
- 场地
- 消课节数
- 权益账户
- 状态
- 取消原因：仅状态为已取消时展示
- 通知状态
- 确认状态：可选
- 备注

不建议弹窗里做：

- 学员资料编辑
- 班次核心字段编辑
- 课包购买
- 会员开卡
- 订场充值
- 完整反馈编辑以外的复杂成长记录

## 6. 联动边界

### 6.1 与班次管理

排课页负责：

- 单次上课记录
- 新增 / 编辑 / 取消课程
- 按有效排课扣减或退回班次课时

班次页负责：

- 班次定义
- 班次学员
- 班次教练
- 班次总课时
- 班次运营判断

边界要求：

- 排课页可以引用班次，但不在排课弹窗里编辑班次核心信息。
- 班次 `usedLessons` 继续由排课驱动。
- 选择多人班次时，当前版本明确提示拆成单人排课。

### 6.2 与学习计划

排课页负责：

- 产生真实上课记录
- 驱动学习计划课时进度

学习计划页负责：

- 展示学员 × 班次的学习跟进状态
- 提供去排课入口

边界要求：

- 学习计划不直接编辑单次排课。
- 排课保存 / 取消后，学习计划展示应同步更新。

### 6.3 与权益账户

排课页负责：

- 选择或推荐本节课使用的权益账户
- 保存时扣权益
- 取消时退权益
- 展示扣课状态摘要

权益账户页负责：

- 权益账户本身的来源、余额、规则、流水明细

边界要求：

- 排课页只展示摘要，不做完整权益流水管理。
- 权益账户为空时必须明确提示不会扣课包权益。
- 暂不做多人排课逐人扣权益。

### 6.4 与订场用户

排课页负责：

- 保存时检查同校区同场地同时间是否已被订场用户占用

订场用户页负责：

- 订场消费、储值、会员、订场历史

边界要求：

- 排课页不编辑订场用户。
- 订场冲突只作为保存拦截或提示。

### 6.5 与课后反馈

排课页负责：

- 打开反馈
- 查看反馈状态
- 已有反馈后保护课程核心字段

反馈负责：

- 教练课后记录
- 体验课转化判断
- 运营跟进建议

边界要求：

- 反馈绑定排课，不脱离排课存在。
- 已反馈排课不能随意改核心事实字段。

### 6.6 与教练运营 / 教练端

排课页是数据来源。

教练运营页负责：

- 教练排课时间轴
- 工作量
- 未反馈
- 跨校区风险

教练端负责：

- 教练看自己的课表
- 填写自己的反馈

边界要求：

- 排课页改状态、时间、教练后，教练运营和教练端必须同步反映。
- 教练端不应看到财务敏感字段，只看必要课包摘要。

## 10. 最终验收标准

完成后，排课表页面应满足：

1. 管理员能用同一套口径筛选 `已排课 / 已结束 / 已取消`。
2. 取消排课和删除误建记录有明确边界。
3. 多人班次当前限制提前提示，不让运营最后一步才失败。
4. 无权益账户排课会明确提示“不会扣课包权益”。
5. 列表或详情能看出扣课状态。
6. 已反馈排课不能再改核心课程事实。
7. 取消排课能记录取消原因。
8. 排课能记录通知状态。
9. 保存前能看到关键确认摘要。
10. 班次、学习计划、权益账户、教练运营、教练端课表联动不被破坏。
